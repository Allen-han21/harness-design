# Run 20260418 v2

## Purpose

이 문서는 `kidsnote_ios` 파일럿 적용 후 받은 피드백을 바탕으로, 장기 실행 하네스 설계를 한 단계 더 구체화한 결과를 기록한다.

이번 라운드의 목적은 두 가지다.

- 파일럿에서 드러난 운영상 경계를 설계 원칙으로 승격하기
- 다음 파일럿부터는 문서 형식만으로도 `검증`과 `승인` 상태가 분명히 읽히게 만들기

## What Changed From v1

`20260418_v1`에서는 방향이 맞는지와 업그레이드 우선순위를 정리했다.

이번 `v2`에서는 그중 실제로 설계 원칙으로 굳혀야 하는 항목을 좁혔다.

1. 루트 `AGENTS.md`는 팀 공용 문서로 남기고, Codex 전용 실행 규칙은 repo-local `.ai/codex-start.md`로 분리한다
2. 모든 active plan은 `Verification Contract`를 포함한다
3. 사람 승인 대기는 별도 상태인 `approval-required`로 남긴다
4. 완료 조건에는 항상 `Human Approval Status`를 포함한다
5. `~/.codex/ai/`는 전역 실행 상태 저장소가 아니라 공통 원칙과 bootstrap reference에 가깝다

## Direction Check

현재 업그레이드 방향은 여전히 본질과 맞다.

이유는 다음과 같다.

- 여전히 중심은 `상태 관리 + 검증 루프 + 종료 규칙`이다
- 멀티에이전트나 외부 오케스트레이터를 늘린 것이 아니라, single-agent loop의 강제력을 높였다
- 문제를 프롬프트 길이로 해결하지 않고 파일 계약과 상태 전이로 해결했다

즉, 이번 변경은 설계 방향 변경이 아니라 `문서 기반 초기형`을 `운영 가능한 규칙형`으로 밀어 올리는 성격이다.

## Key Decisions

### 1. Runtime Entry Must Be Repo-Local

루트 `AGENTS.md`에 Codex 실행 루프를 직접 넣는 방식은 파일럿에선 빨랐지만 운영용으로는 약하다.

이유:

- `AGENTS.md`는 팀 공용 문서일 가능성이 높다
- 프로젝트 기술 가이드와 Codex 전용 실행 규칙이 섞이면 읽기 밀도가 떨어진다
- 다른 에이전트나 사람에게도 불필요한 노이즈가 된다

따라서 실행 진입점은 `.ai/codex-start.md` 같은 repo-local 문서가 맞다.

### 2. Verification Must Become A Contract

기존 `verify`는 명령 목록 수준에 머무르기 쉬웠다.

이제는 plan 단계에서 아래를 먼저 적어야 한다.

- `Automated Checks`
- `Manual Checks`
- `Skipped Checks`
- `Approval Needed`

핵심은 구현 계획이 아니라 `검증 계약`을 먼저 고정하는 것이다.

### 3. Human Approval Needs Its Own State

기술적으로 구현이 끝났더라도 수동 검증이나 제품 판단이 남아 있으면 그건 `completed`가 아니다.

따라서 사람 승인 대기는 별도 상태로 분리해야 한다.

- `blocked`: 기술적 또는 환경적 진행 불가
- `approval-required`: 사람 승인 대기

이 구분이 있어야 `미완료`의 이유가 문서에서 바로 읽힌다.

### 4. Completion Needs Approval Metadata

장기 실행 작업 완료 문서에는 구현/검증 결과만이 아니라 승인 상태가 함께 있어야 한다.

최소 필드:

- `Human Approval Status`
- `approver`
- `remaining checks`

이 필드가 없으면 수동 QA가 남은 작업도 완료처럼 보이기 쉽다.

### 5. Global Path Should Be Reference, Not State

`~/.codex/ai/`는 검토 가치가 있지만 기본 상태 저장소로 쓰는 건 부적절하다.

적합한 역할:

- 공통 원칙
- bootstrap 스크립트
- 재사용 가능한 예시

부적합한 역할:

- active plan
- handoff
- evaluation
- approval 상태
- run artifacts

장기 실행 상태는 `repo`, `task`, `worktree`에 묶여 있기 때문이다.

## Upgrade Items

### 1. `plan`과 `evaluation`의 최소 포맷을 템플릿 수준으로 고정

- reason: 지금은 원칙 문서에는 반영됐지만 예시와 실제 저장소 적용 간 편차가 남을 수 있다
- expected effect: 작업마다 검증 계약과 승인 상태 누락을 줄인다

### 2. `approval-required` 종료 절차 정의

- reason: 승인 완료 후 어떤 파일을 갱신하고 어떤 상태를 정리할지 운영 규칙이 더 필요하다
- expected effect: 수동 승인 이후 상태 전이가 깔끔해진다

### 3. repo bootstrap 예시 추가

- reason: 다른 저장소나 worktree에서 `.ai/` 구조를 재생성하는 시작 비용이 아직 남아 있다
- expected effect: 파일럿을 새 저장소에 옮기기 쉬워진다

### 4. 빠른 verify 스크립트 패턴 문서화

- reason: 여전히 검증은 문서보다 스크립트가 강하다
- expected effect: `Verification Contract`와 실제 실행 명령의 간극을 줄인다

## Success Criteria For Next Round

다음 라운드가 성공이라고 보려면 아래가 확인되어야 한다.

- 새 저장소에서도 `AGENTS.md`를 오염시키지 않고 `.ai/codex-start.md`만으로 시작할 수 있다
