# Round 20260418 v1

## Purpose

이 문서는 `kidsnote_ios`에 Codex 장기실행 하네스를 실제로 붙여 보며 얻은 관찰을 기록한다.

목적은 두 가지다.

- 현재 설계가 본질적인 방향과 같은 방향인지 확인하기
- 다음 라운드에서 무엇을 업그레이드해야 하는지 명확히 하기

## Context

이번 라운드에서 실제로 한 일은 다음과 같다.

- `kidsnote_ios`에 `AGENTS.md` 상단 실행 규칙 추가
- `kidsnote_ios/.ai/` 로컬 상태 저장소 생성
- `plan -> implement -> evaluation -> handoff` 루프를 실제 저장소에 도입
- 실제 이슈 `PK-26719`를 이 루프로 수행

즉, 이번 라운드는 이론 문서 작성이 아니라 `실제 저장소에 단일 Codex 하네스를 붙여 보는 파일럿`이었다.

## Conclusion

현재 방향은 본질적으로 맞다.

핵심 이유는 지금 구조가 아래 세 가지를 중심으로 움직이기 때문이다.

- 상태 관리
- 검증 루프
- 종료 규칙

즉, 멀티에이전트나 외부 오케스트레이터로 먼저 가지 않고, `Codex 단일 에이전트가 오래 안정적으로 일하게 만드는 구조`를 먼저 만들고 있다.

## What Was Correct

이번 라운드에서 방향이 맞다고 판단한 지점은 다음과 같다.

### 1. `AGENTS.md`를 실행 진입점으로 사용했다

긴 설명 문서 대신 시작 시 읽을 순서와 최소 실행 규칙을 상단에 배치했다.

### 2. `.ai/`를 별도 상태 저장소로 분리했다

채팅이나 다른 도구의 흔적이 아니라 repo-local 파일에 상태를 남기기 시작했다.

### 3. `plan -> implement -> evaluation -> handoff` 루프를 실제로 적용했다

구조만 만든 것이 아니라 실제 이슈에 적용해 보았다.

### 4. 작업별 기록을 남기기 시작했다

`index`, `plan`, `evaluation`, `run-log`를 남기면서 재개 가능성의 기반을 만들었다.

### 5. 복잡도를 억제했다

멀티에이전트, 글로벌 메모리, 외부 플랫폼보다 파일 기반 상태와 검증 루프를 우선했다.

## What Is Still Weak

아직은 `문서 기반 초기형`이라 강제력이 약한 부분이 남아 있다.

### 1. `AGENTS.md`가 여전히 길다

상단 실행 규칙은 좋아졌지만, 전체 파일은 여전히 상세 프로젝트 가이드 성격이 강하다.

### 2. `evaluation`이 기계적으로 강제되지는 않는다

지금은 규칙으로 존재하지만 스크립트나 체크 단계로 고정되진 않았다.

### 3. `handoff`는 아직 장시간/다회차 작업에서 충분히 검증되지 않았다

짧은 이슈 적용은 했지만, 세션을 끊었다가 다시 이어가는 시나리오에서 가치가 확인되진 않았다.

### 4. 검증 명령이 문서 중심이다

`swiftlint`, `plutil`, 검색 기반 체크를 수행했지만 검증 루틴이 스크립트로 고정되진 않았다.

### 5. 상태 전이가 더 명확해질 필요가 있다

`active`, `completed`, `blocked` 같은 상태가 파일 구조 수준에서 더 분명해지면 재개성이 좋아진다.

## Upgrade Priorities

다음 업그레이드는 문서를 늘리는 방향이 아니라 `강제력`을 높이는 방향이어야 한다.

### Priority 1. `AGENTS.md`를 더 짧게 만든다

상단 실행 규칙만 남기고, 상세 프로젝트 가이드는 가능한 한 `docs/`나 별도 문서로 분리한다.

목표:

- activation 문서와 reference 문서를 분리
- 시작 시 읽는 문서의 밀도를 높이기

### Priority 2. `verify`를 스크립트로 고정한다

예: `scripts/verify-codex.sh`

이 스크립트는 작업 성격에 따라 최소한 아래를 묶는 방향이 좋다.

- lint
- plist lint
- grep 기반 정책 체크
- 필요 시 빠른 build/manifest 검증

목표:

- 문서 규칙을 실행 명령으로 승격
- 완료 판단의 일관성 확보

### Priority 3. `evaluation` 최소 포맷을 고정한다

매 작업마다 자유서술이 아니라 최소 필드를 유지한다.

권장 최소 필드:

- `target`
- `checks_run`
- `checks_skipped`
- `risks`
- `recommendation`

목표:

- evaluation 품질 편차를 줄이기
- 나중에 회고와 비교가 가능하게 만들기

### Priority 4. `handoff` 의무 조건을 명확히 한다

모든 작업에 handoff를 강제하지 말고, 아래 조건일 때만 의무화하는 편이 좋다.

- 작업이 30분 이상 지속됨
- 작업이 미완료 상태로 끝남
- 세션 종료가 예상됨

목표:

- 과한 문서화 방지
- 정말 필요한 시점에만 상태를 남기기

### Priority 5. `active/completed/blocked` 상태를 더 분명히 한다

현재 구조는 시작점으로는 충분하지만, 상태 전이가 더 분명해질 필요가 있다.

예시:

- `plans/active/`
- `plans/completed/`
- 필요 시 `plans/blocked/`

목표:

- 재개 속도 향상
- 현재 작업 상태를 파일 구조만으로 파악 가능하게 만들기

### Priority 6. `run-log`는 사건 로그만 남긴다

`run-log`는 설명 문서가 아니라 사건 로그여야 한다.

권장 형식:

- 무엇을 결정했는가
- 왜 그렇게 결정했는가
- 다음 작업에 어떤 영향을 주는가

목표:

- 기록 비용 최소화
- 장기 작업에서 의미 있는 변화만 남기기

### Priority 7. 실제 장기 작업으로 검증한다

지금까지는 `하네스 부착 + 단기 이슈 적용` 단계다.

다음 라운드의 핵심 검증 질문은 이것이다.

- 세션을 끊었다가 다시 이어도 빠르게 복구되는가
- handoff만 읽고 바로 작업을 재개할 수 있는가
- 완료 착각과 검증 누락이 실제로 줄어드는가
