# Claude Code Adapter

## 목적

이 문서는 `Claude Code`에서 공통 하네스 철학을 실제 작업 흐름에 맞게 적용하는 방법을 정리한다.

## 역할

Claude Code adapter는 다음을 다룬다.

- 어떤 문서를 세션 초기에 안정적으로 노출할지
- 어떤 로컬 작업 문서를 유지할지
- Claude Code의 긴 문맥 활용과 압축 특성을 어떻게 다룰지

## 기본 진입점

Claude Code에서도 가장 중요한 것은 현재 저장소 안의 실행 문서다.

권장 우선순위:

1. 현재 프로젝트 루트
2. 프로젝트의 실행 규칙 문서
3. 로컬 작업 상태 문서
4. 코드, 테스트, 실행 결과

## 권장 적용 방식

- 공통 철학은 `harness-design`에 둔다
- 프로젝트 안에는 실행에 필요한 요약 문서만 둔다
- 긴 문맥에 기대기보다 현재 작업 상태를 구조화된 문서로 남긴다

## Claude Code에서 잘 작동하는 방식

- 세션 간 이어받기 쉬운 handoff 문서
- 길게 설명하기보다 구조화된 상태 요약
- plan과 evaluation이 분리된 문서 구조
- 문서와 실제 코드 상태가 함께 검증되는 흐름

## 주의할 점

- 긴 문맥을 사용할 수 있어도, 그것만으로 안정적인 운영이 되지는 않는다
- 과거 대화에 과하게 의존하면 상태 추적이 흐려질 수 있다
- 로컬 문서 없이 세션 기억만 믿는 방식은 재현성이 낮다

## 권장 최소 적용 세트

- 실행 요약 문서 1개
- `.ai/plans/`
- `.ai/handoffs/`
- `.ai/evaluations/`
- 실제 검증 결과를 남길 수 있는 최소 루프

## Claude Code에서 adapter가 다뤄야 할 것

- 어떤 문서를 세션 시작 요약으로 둘지
- handoff 문서 형식을 얼마나 구조화할지
- 긴 문맥 활용과 파일 기반 상태 관리의 균형

## Claude Code에서 adapter가 다루지 말아야 할 것

- 공통 원칙의 중복 정의
- 저장소 구조 원칙 자체의 변경
- 도구 특성과 무관한 조직 정책의 재설계
