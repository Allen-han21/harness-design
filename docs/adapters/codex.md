# Codex Adapter

## 목적

이 문서는 `Codex`에서 공통 하네스 철학을 실제 실행 흐름에 연결하는 방법을 정리한다.

## 역할

Codex adapter는 다음 문제를 다룬다.

- 현재 저장소 안에서 무엇을 기본 진입점으로 둘지
- 어떤 문서가 실제 세션에서 먼저 읽히도록 할지
- 어떤 규칙은 문서가 아니라 기계적으로 강제해야 하는지

## 기본 진입점

Codex에서 가장 중요한 진입점은 보통 다음 순서다.

1. 현재 프로젝트 루트
2. `AGENTS.md`
3. 현재 작업에 가까운 로컬 문서
4. 실제 코드와 테스트 결과

따라서 Codex에서는 `harness-design` 원문보다 현재 프로젝트 안의 짧은 실행 문서가 더 중요하다.

## 권장 적용 방식

### 저장소 바깥의 원칙

- 공통 철학은 `harness-design`에 보관한다

### 저장소 안의 실행층

- `AGENTS.md`에 현재 프로젝트용 실행 요약을 둔다
- 로컬 작업 문서는 `<repo>/.ai/`에 둔다
- 템플릿은 필요 시 프로젝트 안으로 복사해 사용한다

## Codex에서 잘 작동하는 방식

- 짧고 구체적인 지시
- 현재 저장소 안에 있는 문서
- 파일 경로가 명확한 템플릿
- 테스트, lint, 스크립트처럼 기계적으로 확인 가능한 종료 조건

## 주의할 점

- `harness-design` 저장소에 문서가 있다고 해서 자동으로 반영되지는 않는다
- 긴 정책 문서를 매번 읽게 하는 방식은 불안정하다
- 경로만 알려주는 방식은 activation이 약하다

## 권장 최소 적용 세트

- 프로젝트 루트의 `AGENTS.md`
- 로컬 `.ai/`
- `plan`, `handoff`, `evaluation` 템플릿
- 최소 하나 이상의 기계적 검증 수단

## Codex에서 adapter가 다뤄야 할 것

- 어떤 규칙을 `AGENTS.md`로 노출할지
- 어떤 템플릿을 로컬로 복사할지
- 어떤 검증 루프를 종료 조건으로 둘지

## Codex에서 adapter가 다루지 말아야 할 것

- 공통 철학 자체의 재정의
- 도구와 무관한 문서 수명주기 규칙의 중복 정의
- 모든 프로젝트에 동일한 상세 워크플로 강제
