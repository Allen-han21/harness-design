# Gemini Adapter

## 목적

이 문서는 `Gemini` 계열 도구에서 공통 하네스 철학을 어떻게 연결할지 정리한다.

## 역할

Gemini adapter는 다음을 다룬다.

- 어떤 문서를 얇은 activation layer로 둘지
- 어떤 구조가 탐색과 이해에 유리한지
- 공통 원칙을 프로젝트 실행 규칙으로 어떻게 내려줄지

## 기본 진입점

Gemini에서도 공통 철학 저장소보다 현재 프로젝트 안의 실행 문서가 우선이다.

권장 우선순위:

1. 현재 작업 중인 프로젝트 루트
2. 프로젝트 실행 규칙 문서
3. 로컬 상태 문서
4. 코드, 로그, 테스트, 스크립트

## 권장 적용 방식

- 공통 철학은 `harness-design`에서 관리한다
- 프로젝트에는 짧고 명시적인 실행 문서를 둔다
- 로컬 상태는 `.ai/`에서 관리한다
- 템플릿은 프로젝트 문맥에 맞게 얇게 복사해 쓴다

## Gemini에서 잘 작동하는 방식

- 구조가 명확한 문서
- 파일 경로와 역할이 분명한 저장소
- 긴 설명보다 체크리스트형 실행 규칙
- 문서와 검증 스크립트가 함께 존재하는 상태

## 주의할 점

- 경로만 알려주는 방식은 기대만큼 강한 activation을 만들지 못할 수 있다
- 공통 저장소 원문을 항상 직접 참조하게 하는 구조는 운영 비용이 커질 수 있다
- 프로젝트 안의 실행층이 없으면 원하는 행동 일관성이 낮아질 수 있다

## 권장 최소 적용 세트

- 프로젝트 실행 요약 문서
- `.ai/` 작업 공간
- 공통 템플릿의 로컬 복사본
- 검증 기준이 보이는 체크리스트 또는 스크립트

## Gemini에서 adapter가 다뤄야 할 것

- 어떤 실행 문서를 가장 앞단에 둘지
- 어떤 문서 구조가 탐색성과 적용성에 유리한지
- 템플릿을 얼마나 얇게 현지화할지

## Gemini에서 adapter가 다루지 말아야 할 것

- 공통 철학의 별도 버전 만들기
- 프로젝트별 세부 운영을 도구 문서 안에 과하게 넣는 것
- 실제 검증 장치 없이 문서만으로 해결하려는 것
