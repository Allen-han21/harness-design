# Document Language Policy

## 목적

이 문서는 하네스 설계 문서의 언어를 정할 때 다음 세 가지를 함께 만족시키기 위한 정책이다.

- 낮은 유지보수 비용
- 작성자 기준의 높은 수정 용이성
- 도구와 저장소를 넘는 적절한 이식성

## 기본 입장

기본값으로 이중 언어 문서 체계를 운영하지 않는다.

대부분의 살아있는 문서는 하나의 언어로 유지하고, 구조의 이식성은 영어 기반의 이름 규칙으로 확보한다.

이 저장소의 기본 권장안은 다음과 같다.

- 문서 내용: 살아있는 문서는 한국어 우선
- 폴더명과 파일명: 영어
- 템플릿 키와 반복되는 구조 라벨: 영어
- 장기 자산 또는 외부 공개 문서: 비용 대비 가치가 있을 때 영어

## 배경

영어는 도구와 저장소를 넘는 재사용성 측면에서 유리하지만, 모든 문서를 영어로 강제하면 수정 비용이 커진다.

하네스 문서는 계속 갱신되어야만 의미가 있다. 이식성은 높지만 거의 수정되지 않는 문서보다, 자주 손볼 수 있는 문서가 운영상 더 가치 있다.

이 정책의 목표는 "영어 비중 최대화"가 아니다. 목표는 "구조적 이식성을 잃지 않으면서 유지보수 비용을 가장 낮추는 것"이다.

## 언어 계층

### 1. 살아있는 문서

자주 수정되는 문서는 유지보수 속도를 우선해야 한다.

권장 언어: 한국어

예시:

- 작업 메모
- 계획 문서
- handoff 문서
- 평가 문서
- 회고
- 로컬 운영 규칙

### 2. 구조적 인터페이스

에이전트가 여러 저장소에서 일관되게 탐색할 수 있어야 하는 부분은 안정적으로 유지한다.

권장 언어: 영어

예시:

- `plans`, `handoffs`, `evaluations`, `templates` 같은 폴더명
- `plan-template.md` 같은 파일명
- 템플릿에서 반복적으로 쓰는 안정적인 섹션명이나 키

권장 라벨 예시:

- `Purpose`
- `Scope`
- `Constraints`
- `Done Criteria`
- `Open Questions`

### 3. 장기 재사용 자산

여러 프로젝트에서 재사용되는 자산은 실제 재사용 가치가 확인된 뒤 영어로 승격할 수 있다.

예시:

- 핵심 원칙 문서
- 저장소 README
- 재사용 템플릿
- 일반화된 정책 문서

## 기본 규칙

명확한 이유가 없으면 `내용은 한국어`, `구조는 영어`를 기본으로 한다.

즉, 다음을 의미한다.

- 설명과 판단은 한국어로 작성한다
- 디렉터리명과 파일명은 영어로 유지한다
- 재사용 템플릿의 섹션 구조는 안정적으로 유지한다

## 피해야 할 것

다음은 기본값으로 삼지 않는다.

- 모든 문서를 한국어와 영어 두 벌로 유지하는 것
- 임시 메모까지 영어로 번역하는 것
- 초기 초안부터 영어로 강제하는 것
- 이유 없이 디렉터리마다 이름 규칙을 다르게 가져가는 것

이런 방식은 하네스 품질을 크게 높이지 못하는데도 유지보수 비용만 증가시킨다.

## 승격 규칙

한국어로 작성된 작업 문서가 여러 프로젝트에서 재사용될 가치가 생기면 영어 자산으로 승격할지 판단한다.

다음 중 하나라도 만족하면 영어 승격을 검토한다.

- 여러 저장소에서 재사용될 가능성이 있다
- 표준 템플릿이 될 수 있다
- 외부 공개 문서가 될 가능성이 높다
- 도구 중립적인 이식성이 실제로 중요하다

해당되지 않으면 한국어로 유지한다.

## 운영 요약

- 살아있는 내용은 수정하기 쉬워야 한다
- 구조는 재사용하기 쉬워야 한다
- 이중 언어 유지보수는 기본값이 아니다
- 영어는 보조 수단이지 보편 강제가 아니다
